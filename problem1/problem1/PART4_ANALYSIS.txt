PART 4: DESIGN ANALYSIS & RECOMMENDATIONS
Cache Hierarchy Optimization using gem5

Based on the comprehensive parameter sweep of 108 configurations using a 
128×128 matrix multiplication benchmark on RISC-V gem5 simulator.

a) Performance Bottlenecks

Is execution time dominated by L1D misses, L2 misses, or memory stalls?

Analysis of the simulation results reveals that execution time is HEAVILY 
DOMINATED BY L2 MISSES AND MEMORY STALLS. The data shows:

- L1D hit rate remains consistently low (~50%) across all configurations due 
  to the 192KB working set far exceeding all L1D cache sizes (16-64KB). This 
  creates massive L1D miss traffic flowing to L2.

- L2 hit rate shows DRAMATIC variation (0.005% to 99.7%), directly correlating 
  with execution time. Configurations with small L2 caches (128kB-256kB) suffer 
  catastrophic performance degradation, with execution times reaching 1,974B 
  ticks compared to 684B ticks for 1MB L2 - a 2.9x difference!

- The 512kB L2 configurations show intermediate performance (~750B-800B ticks), 
  indicating L2 capacity misses are the dominant bottleneck when L2 is too small.

What percentage of memory requests reach main memory?

Calculating from the best and worst cases:
- Best case (64kB L1D, 1MB L2, 8-way assoc): L1D miss rate ~50%, L2 hit rate 
  ~99.7% → Only 0.15% of ALL memory requests reach main memory (50% × 0.3%)
- Worst case (16kB L1D, 128kB L2, 2-way assoc): L1D miss rate ~50%, L2 hit 
  rate ~0.01% → Nearly 50% of ALL requests hit main memory!

This 333x difference in main memory traffic explains the catastrophic 
performance gap between configurations.

b) Cache Efficiency

Which cache level has the best hit rate? Why?

L1I (Instruction Cache) has the BEST hit rate at 99.999% across all 
configurations. This is expected because:
- Matrix multiplication has a small, tight inner loop
- Code footprint is minimal (~few KB)
- High temporal locality in instruction fetches
- Even 16kB L1I is massively oversized for this workload

However, for DATA caches, L2 demonstrates superior efficiency when properly 
sized. The 1MB L2 achieves 99.7% hit rate despite receiving 50% of all memory 
requests from L1D misses, while L1D is fundamentally limited by capacity.

Is L2 size or associativity more important?

L2 SIZE is DRAMATICALLY more important than associativity. Evidence:

Size Impact:
- 128kB→256kB L2: 75% reduction in execution time average
- 256kB→512kB L2: 60% reduction in execution time  
- 512kB→1MB L2: 15% additional improvement

Associativity Impact (holding size constant):
- 4-way→8-way L2: <2% performance difference
- 8-way→16-way L2: <1% performance difference

The matrix multiplication exhibits streaming access patterns with poor spatial 
locality, making CAPACITY the critical factor. Associativity helps with 
conflict misses, but the working set size dominates performance.

c) Cost-Benefit Trade-off

What's the smallest L1D+L2 configuration that achieves 90% of peak performance?

Peak performance: 684B ticks (64kB L1D, 1MB L2, 8-way assoc)
90% performance target: 756B ticks or better

Optimal configuration achieving this:
- L1D: 32kB (2-way or 4-way associativity)
- L2: 512kB (4-way associativity)
- Total cache: 544 KB
- Execution time: ~750B ticks (90.9% of peak)

This provides excellent cost-benefit ratio - using only 53% of the cache 
capacity of the peak configuration (32+512 vs 64+1024 KB) while maintaining 
90% performance.

How much performance do you lose by using direct-mapped caches (assoc=1)?

Note: Our sweep tested associativities of 2, 4, 8, 16 but NOT direct-mapped 
(assoc=1). However, extrapolating from the minimal impact of associativity 
increases:
- 2-way→4-way: ~1-3% improvement
- 4-way→8-way: ~1-2% improvement

We estimate direct-mapped (1-way) caches would lose approximately 5-10% 
performance compared to 2-way associative caches, primarily due to conflict 
misses in the L2 cache during matrix traversals. For L1D, the loss would be 
minimal since it's already thrashing due to capacity constraints.

d) Design Recommendations

Scenario 1: POWER-CONSTRAINED SYSTEM (minimize cache size)

Recommended Configuration:
- L1I: 16kB, 2-way associative
- L1D: 16kB, 2-way associative  
- L2: 512kB, 4-way associative
Total Cache: 544 KB
Expected Performance: ~750-800B ticks

Justification: Minimizes L1 caches since they provide limited benefit for this 
workload. Invests area budget in L2 where capacity matters most. The 512kB L2 
is the minimum size achieving acceptable performance, capturing most of the 
working set's temporal locality. 2-way/4-way associativity provides good 
conflict miss reduction without excessive area overhead.

Scenario 2: HIGH-PERFORMANCE SYSTEM (maximize performance)

Recommended Configuration:
- L1I: 16kB, 4-way associative
- L1D: 64kB, 8-way associative
- L2: 1MB, 4-way associative  
Total Cache: 1,104 KB
Expected Performance: 684B ticks (BEST)

Justification: Maximizes L1D and L2 cache sizes to minimize miss rates. The 
64kB L1D provides the highest hit rate possible given the working set. 1MB L2 
achieves 99.7% hit rate, eliminating nearly all main memory traffic. 8-way 
associativity in L1D and 4-way in L2 provide diminishing returns beyond these 
points, so higher associativity is not cost-effective.

Scenario 3: BALANCED SYSTEM (best performance/cost ratio)

Recommended Configuration:
- L1I: 16kB, 2-way associative
- L1D: 32kB, 4-way associative
- L2: 512kB, 8-way associative
Total Cache: 560 KB  
Expected Performance: ~750B ticks (90.9% of peak)

Justification: This configuration achieves the optimal "knee of the curve" - 
90% of peak performance with 51% of the cache capacity. The 32kB L1D provides 
meaningful improvement over 16kB without excessive area. 512kB L2 is the 
sweet spot where further increases yield diminishing returns. 8-way L2 
associativity provides excellent conflict miss coverage without the area 
overhead of 16-way.

Performance-per-KB metric: This configuration delivers 1.34x better 
performance-per-kilobyte than the high-performance configuration, making it 
ideal for mainstream applications where both performance and cost matter.

# PARETO-OPTIMAL CONFIGURATIONS ANALYSIS

What are Pareto-optimal configurations?

Pareto-optimal configurations represent the "efficiency frontier" - these are 
cache designs where you CANNOT improve one performance metric without making 
another metric worse. In our analysis, we examine the trade-off between:
- Execution time (lower is better)
- L1D hit rate (higher is better)

A configuration is Pareto-optimal if:
1. No other configuration has both better execution time AND better hit rate
2. Improving one metric requires sacrificing the other
3. It represents a fundamental design trade-off point

Identified Pareto-Optimal Configurations:

From our 108-configuration sweep, the Pareto frontier includes configurations 
with 64kB L1D caches paired with various L2 sizes. These represent the optimal 
trade-offs because:

1. 64kB L1D maximizes L1D hit rate (~50.04% - the theoretical maximum given 
   the 192KB working set)
2. Varying L2 size trades execution time for cache capacity
3. Smaller L1D configurations (16kB, 32kB) are strictly dominated - they have 
   BOTH worse hit rates AND worse (or equal) execution times

Pareto Frontier Insights:

- Moving along the frontier: Larger L2 caches (512kB→1MB) reduce execution 
  time but increase cost/area
- Off-frontier configs: 16kB and 32kB L1D configurations are suboptimal - 
  they could improve hit rate by increasing to 64kB without hurting execution 
  time
- Design choice: Select a point on the Pareto frontier based on your 
  constraints (cost vs performance)

Practical Application:

The Pareto frontier plot directly guides the three design scenarios:
- Power-constrained: Select leftmost Pareto point (smaller cache, accept 
  lower hit rate)
- High-performance: Select rightmost Pareto point (largest cache, best hit 
  rate)
- Balanced: Select middle Pareto point (optimal performance-per-cost)

All configurations NOT on the Pareto frontier are strictly inferior design 
choices and should be avoided.
